# Story 2.2: Card Creation with Color Selection

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a user,
I want to create new cards by clicking the sticky-note pad or [+] button and choosing a color before I type,
So that each card feels like placing a real post-it note — color is an intentional first gesture, not an afterthought.

## Acceptance Criteria

1. **Given** the board is visible and the user hovers over the `CardCreationPad` in the To Do column **When** the hover state is active **Then** a dog-ear peel animation is visible on the pad corner (CSS transition — no JavaScript-driven animation)

2. **Given** the user clicks the `CardCreationPad` **When** the color picker opens **Then** exactly 6 color swatches are displayed: yellow, pink, blue, green, orange, purple

3. **Given** the color picker is open **When** the user clicks a color swatch **Then** the color picker closes and a new inline card creation form appears at the top of the To Do column with the selected color as its background

4. **Given** a new card creation form is visible **When** it appears **Then** the card title `<input>` field is automatically focused so the user can begin typing without any additional click

5. **Given** the [+] button is visible in the To Do column **When** clicked **Then** it triggers the same card creation flow as the `CardCreationPad` — the color picker opens first, then the form appears on color selection with focused title input

6. **Given** a card title has been entered and the user clicks elsewhere (blur) **When** the blur event fires **Then** the card title is committed and a new card is persisted via `dispatch({ type: 'ADD_CARD', payload: { card, columnId: 'todo' } })`

7. **Given** a card title is left completely empty when blurred **When** the blur event fires **Then** no card is added to the board — creation is silently cancelled, the form disappears, and no `ADD_CARD` is dispatched

8. **Given** a new card is committed **When** its `id` is inspected **Then** it is a valid UUID generated by `crypto.randomUUID()` — never `Math.random()` or `Date.now()` alone

9. **Given** a new card is committed **When** the board state is inspected **Then** the card appears as the first item (prepended) in the `todo` column array, consistent with how `ADD_CARD` is implemented in `BoardReducer`

10. **Given** the board contains at least one card **When** the Board renders **Then** each column reads its card IDs from `boardState.columns[columnId]`, looks up `boardState.cards[id]`, and renders a minimal `Card` component per card — Board.tsx is the first component to consume `useBoardContext()`

## Dev Notes

### Escape Key Cancel — Preventing Blur from Dispatching After Escape

When the user presses `Escape`, we want to cancel creation without saving. The problem: pressing Escape on an input causes a blur event to fire immediately after the keydown. If the blur handler checks `value.trim()`, and the user typed something before pressing Escape, the blur would save the card.

**Solution pattern using a ref flag:**
```tsx
const cancelledRef = useRef(false);

function handleTitleKeyDown(e: React.KeyboardEvent<HTMLInputElement>) {
  if (e.key === 'Escape') {
    cancelledRef.current = true;
    setPhase('idle'); // triggers blur but cancelledRef is already set
  } else if (e.key === 'Enter') {
    e.currentTarget.blur(); // blur will handle dispatch
  }
}

function handleTitleBlur(e: React.FocusEvent<HTMLInputElement>) {
  if (cancelledRef.current) {
    cancelledRef.current = false;
    setPhase('idle');
    return;
  }
  const title = e.currentTarget.value.trim();
  if (title) {
    const card: Card = {
      id: crypto.randomUUID(),
      title,
      description: '',
      color: selectedColor,
      createdAt: Date.now(),
    };
    dispatch({ type: 'ADD_CARD', payload: { card, columnId: 'todo' } });
  }
  setPhase('idle');
}
```

### Auto-Focus Pattern

Use a `useEffect` that watches the `phase` state:
```tsx
const titleInputRef = useRef<HTMLInputElement>(null);
useEffect(() => {
  if (phase === 'card-creating') {
    titleInputRef.current?.focus();
  }
}, [phase]);
```

### Color Class vs. Inline Style

The architecture says "use CSS custom properties for all visual values — no hardcoded hex colors." For the card background color where the color name is dynamic (comes from `CardColor` type), the approved approach is `style={{ backgroundColor: 'var(--color-card-yellow)' }}` — this references the design token at runtime via a dynamic CSS variable reference string. This is NOT hardcoding a hex value; it evaluates at render time using the CSS custom property.

Do NOT use a class map approach like `.card-yellow { background-color: #FFE135 }` — that would hardcode the hex value in CSS.

### Existing `BoardReducer` `ADD_CARD` Behavior

From Story 2.1, `ADD_CARD` is already implemented in `BoardReducer.ts` as:
- `cards: { ...state.cards, [action.payload.card.id]: action.payload.card }`
- Prepends the card ID to the target column: `[action.payload.card.id, ...state.columns[action.payload.columnId]]`

This means new cards always appear at the TOP of the To Do column. **Do not change this behavior.**

### `BoardState.columns` Key Mapping

The column IDs are `'todo'`, `'inProgress'`, `'done'` (camelCase). The `BoardState.columns` shape is:
```typescript
columns: {
  todo: string[];
  inProgress: string[];
  done: string[];
}
```
Access pattern: `boardState.columns[col.id]` — `ColumnId` is `'todo' | 'inProgress' | 'done'`.

### Board.tsx — Column Card Computation

Replace the static `<Column>` mapping in `Board.tsx` with one that maps card IDs to actual `Card` objects. Use a clean pre-filter:
```tsx
const { boardState } = useBoardContext();

// In JSX:
{COLUMNS.map((col) => {
  const columnCards = boardState.columns[col.id]
    .map(id => boardState.cards[id])
    .filter((c): c is Card => Boolean(c));
  return (
    <Column
      key={col.id}
      id={col.id}
      title={col.title}
      emptyStateText={col.emptyStateText}
      cards={columnCards}
    />
  );
})}
```

The `.filter((c): c is Card => Boolean(c))` type guard safely handles any race condition where a card ID exists in a column array but the card itself has been removed.
